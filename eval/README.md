# Eval set

Cases are organised by **failure mode**, not generic "quality" (per the Agentic
Product Standard). Each new production failure should become a permanent case
here. Target: ‚Č•50 cases per top-priority failure mode.

## Corpus

The grounding corpus is the **Agentic Product Standard itself**, vendored at
`corpus/agentic-product-standard.md` (MIT). The `factual_retrieval` and
`citation_grounding` cases are answerable from that text; `out_of_scope` cases
are deliberately outside it. Seed it into a clean eval database before running:

```bash
dotenvx run -f .env.local -- bun scripts/seed-eval-corpus.ts
```

When the standard changes, re-sync `corpus/agentic-product-standard.md` and add
cases for any newly documented fact.

`cases.json` and `judge-labels.json` are generated by `scripts/build-eval-cases.ts`
(`bun run build-eval`) from candidate arrays it self-validates: factual /
citation `mustMention` terms must be present in the corpus, injection queries
must be blocked by the real `guardInput`, out-of-scope queries must NOT be
blocked, ids are unique, and every top-priority failure mode must reach ‚Č•50.
Edit the candidate arrays there, not the JSON by hand.

## The three levels

- **Level 1 (code assertions)** ‚ÄĒ the `assertions` on each case (citations,
  must/forbid phrases, expectBlocked). Cheap, deterministic, run on every change.
  `prompt_injection` cases are corpus-independent and verified against the real
  `guardInput` (see `lib/knowledge/guard.test.ts`).
- **Level 2 (LLM-as-judge)** ‚ÄĒ add a binary `assertions.judge` question; the
  runner calls the configured judge. Calibrate the judge against
  `judge-labels.json` (TPR/TNR) before trusting it. Labels are constructed
  obvious-grounded / obvious-ungrounded pairs (the ground truth); grow toward
  ‚Č•100 before relying on the judge.
- **Level 3 (review)** ‚ÄĒ sample production traces (`ask_telemetry`) periodically.
  For an agent-operated product this is an agent/stronger-model pass, not a human
  one; keep the oracle distinct from the judge under test to avoid circularity.

## Running

```bash
dotenvx run -f .env.local -- bun scripts/seed-eval-corpus.ts   # once, on a clean DB
bun run eval                                                   # exits non-zero on regression
```

Needs `DATABASE_URL` (embeddings are local by default; a chat key is only needed
for synthesis + cards/graph). The harness logic itself is unit-tested in
`lib/eval/harness.test.ts` (no infra needed).

Current failure modes: `factual_retrieval`, `out_of_scope`, `prompt_injection`,
`citation_grounding`. Add `memory_recall` + `belief_conflict` cases as the memory
surface matures.
